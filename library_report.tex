\chapter{Introduction}

\section{Project Overview}

This report documents the design, implementation, and demonstration of a Library Management
System built as the final project for the Introduction to Databases course at the National
Economics University. The system uses MySQL as the relational database engine and Python as
the application programming layer.

A library management system is a practical and well-defined domain for demonstrating core
database concepts: normalized table design, referential integrity, transactional operations, and
automated behavior via triggers and stored procedures. The project covers the full lifecycle from
schema design to a working console application.

The source code is organized as follows:
\begin{itemize}
  \item \texttt{database/finaldb.sql} -- Full SQL script: schema, sample data, indexes,
        views, stored procedures, triggers, and UDF.
  \item \texttt{src/main.py} -- Python CLI application.
\end{itemize}

\section{Objectives}

\begin{itemize}
  \item Design a fully normalized relational database schema that covers books, authors,
        categories, readers, and borrowing transactions.
  \item Implement advanced database objects: indexes, views, stored procedures, triggers,
        and user-defined functions.
  \item Develop a Python CLI application that exposes all core functionalities through an
        interactive menu system.
  \item Enforce data integrity at the database level through constraints, foreign keys,
        and triggers.
  \item Apply basic database security and administration principles.
\end{itemize}

\section{Functional Requirements}

\subsection*{Book Management}
\begin{itemize}
  \item View the full book catalog with author and category details.
  \item Add new books with confirmation before inserting.
  \item Edit book quantity.
  \item Delete a book (with confirmation and FK protection).
  \item Search books by name using partial keyword matching (\texttt{LIKE}).
\end{itemize}

\subsection*{Reader Management}
\begin{itemize}
  \item View all registered readers.
  \item Register a new reader (name, address, phone).
  \item Update reader address and phone number.
\end{itemize}

\subsection*{Author \& Category Management}
\begin{itemize}
  \item View, add, and delete authors.
  \item View, add, and delete categories.
\end{itemize}

\subsection*{Borrowing \& Returning}
\begin{itemize}
  \item Borrow a book: validates reader and book existence, shows available quantity,
        calls stored procedure \texttt{BorrowBook}.
  \item Return a book: shows borrow details and days borrowed, warns if overdue,
        calls stored procedure \texttt{ReturnBook}.
  \item Prevents returning an already-returned record.
\end{itemize}

\subsection*{Reports}
\begin{itemize}
  \item Full borrowing report showing all transactions with status.
  \item Overdue report using UDF \texttt{fn\_DaysOverdue} to calculate days borrowed
        and filter records exceeding 7 days.
  \item Statistical summary: total books, readers, and borrowing counts.
  \item Top 5 most borrowed books.
\end{itemize}

\section{Indexes}

\begin{lstlisting}[style=sqlstyle]
CREATE INDEX idx_bookname ON Books(BookName);

\end{lstlisting}

\texttt{idx\_bookname} accelerates \texttt{LIKE}-based keyword searches in the

displays book details.

\section{Views}

\subsection{View\_BookDetails}

\begin{lstlisting}[style=sqlstyle, caption={View\_BookDetails}]
CREATE VIEW View_BookDetails AS
SELECT b.BookID, b.BookName, a.AuthorName,
       c.CategoryName, b.Quantity
FROM Books b
JOIN Authors    a ON b.AuthorID   = a.AuthorID
JOIN Categories c ON b.CategoryID = c.CategoryID;
\end{lstlisting}

\subsection{View\_BorrowingReport}

\begin{lstlisting}[style=sqlstyle, caption={View\_BorrowingReport}]
CREATE VIEW View_BorrowingReport AS
SELECT r.ReaderName, b.BookName,
       br.BorrowDate, br.ReturnDate, br.Status
FROM Borrowing br
JOIN Readers r ON br.ReaderID = r.ReaderID
JOIN Books   b ON br.BookID   = b.BookID;
\end{lstlisting}

\subsection{View\_OverdueReport}

This view calls \texttt{fn\_DaysOverdue()} to ensure the overdue calculation
is consistent with the UDF used in the Python application.

\begin{lstlisting}[style=sqlstyle, caption={View\_OverdueReport}]
CREATE VIEW View_OverdueReport AS
SELECT r.ReaderName, b.BookName, br.BorrowDate,
       fn_DaysOverdue(br.BorrowID) AS DaysOverdue
FROM Borrowing br
JOIN Readers r ON br.ReaderID = r.ReaderID
JOIN Books   b ON br.BookID   = b.BookID
WHERE br.Status = 'Borrowed'
  AND fn_DaysOverdue(br.BorrowID) > 7;
\end{lstlisting}

\section{Stored Procedures}

\subsection{BorrowBook(p\_ReaderID, p\_BookID)}

\begin{lstlisting}[style=sqlstyle, caption={Stored Procedure: BorrowBook}]
CREATE PROCEDURE BorrowBook(IN p_ReaderID INT, IN p_BookID INT)
BEGIN
  DECLARE book_qty INT;
  SELECT Quantity INTO book_qty FROM Books WHERE BookID = p_BookID;
  IF book_qty > 0 THEN
    INSERT INTO Borrowing (ReaderID, BookID, BorrowDate, Status)
    VALUES (p_ReaderID, p_BookID, NOW(), 'Borrowed');
  ELSE
    SIGNAL SQLSTATE '45000'
      SET MESSAGE_TEXT = 'Book is out of stock';
  END IF;
END
\end{lstlisting}

\subsection{ReturnBook(p\_BorrowID)}

\begin{lstlisting}[style=sqlstyle, caption={Stored Procedure: ReturnBook}]
CREATE PROCEDURE ReturnBook(IN p_BorrowID INT)
BEGIN
  UPDATE Borrowing
  SET Status = 'Returned', ReturnDate = NOW()
  WHERE BorrowID = p_BorrowID;
END
\end{lstlisting}

\section{Triggers}

\subsection{trg\_reduce\_quantity}

Fires \texttt{AFTER INSERT} on \texttt{Borrowing} and decrements the book's
\texttt{Quantity} by 1 automatically.

\begin{lstlisting}[style=sqlstyle, caption={Trigger: trg\_reduce\_quantity}]
CREATE TRIGGER trg_reduce_quantity
AFTER INSERT ON Borrowing FOR EACH ROW
BEGIN
  UPDATE Books SET Quantity = Quantity - 1
  WHERE BookID = NEW.BookID;
END
\end{lstlisting}

\subsection{trg\_increase\_quantity}

Fires \texttt{AFTER UPDATE} on \texttt{Borrowing} and increments \texttt{Quantity}
by 1 when the status changes to \texttt{'Returned'}.

\begin{lstlisting}[style=sqlstyle, caption={Trigger: trg\_increase\_quantity}]
CREATE TRIGGER trg_increase_quantity
AFTER UPDATE ON Borrowing FOR EACH ROW
BEGIN
  IF NEW.Status = 'Returned' THEN
    UPDATE Books SET Quantity = Quantity + 1
    WHERE BookID = NEW.BookID;
  END IF;
END
\end{lstlisting}

\section{User-Defined Function}

\subsection{fn\_DaysOverdue(p\_BorrowID)}

Returns the number of days elapsed since the borrow date for any record
still in \texttt{'Borrowed'} status. Returns 0 if the record has already
been returned or the \texttt{BorrowID} is invalid.

\begin{lstlisting}[style=sqlstyle, caption={UDF: fn\_DaysOverdue}]
CREATE FUNCTION fn_DaysOverdue(p_BorrowID INT)
RETURNS INT
NOT DETERMINISTIC
READS SQL DATA
BEGIN
  DECLARE borrow_dt  DATETIME;
  DECLARE days_diff  INT;
  SELECT BorrowDate INTO borrow_dt
  FROM Borrowing
  WHERE BorrowID = p_BorrowID AND Status = 'Borrowed';
  IF borrow_dt IS NULL THEN RETURN 0; END IF;
  SET days_diff = DATEDIFF(NOW(), borrow_dt);
  RETURN days_diff;
END
\end{lstlisting}

Usage in \texttt{src/main.py} (function \texttt{overdue\_with\_udf()}):

\begin{lstlisting}[style=sqlstyle]
SELECT r.ReaderName, b.BookName, br.BorrowDate,
       fn_DaysOverdue(br.BorrowID) AS DaysOverdue
FROM Borrowing br
JOIN Readers r ON br.ReaderID = r.ReaderID
JOIN Books   b ON br.BookID   = b.BookID
WHERE br.Status = 'Borrowed'
  AND fn_DaysOverdue(br.BorrowID) > 7
ORDER BY DaysOverdue DESC;
\end{lstlisting}

\section{Data Security Measures}

\begin{itemize}
  \item \textbf{SQL Injection Prevention:} All queries in \texttt{src/main.py} use
        parameterized statements (\texttt{\%s} placeholders), never string concatenation.
  \item \textbf{Domain Integrity:} CHECK constraints enforce
        \texttt{Quantity >= 0} and \texttt{Status IN ('Borrowed','Returned')}.
  \item \textbf{Referential Integrity:} \texttt{ON DELETE RESTRICT} prevents deletion
        of authors or categories still referenced by books.
  \item \textbf{Uniqueness:} UNIQUE constraints on \texttt{CategoryName} and
        \texttt{PhoneNumber} prevent duplicate entries.
  \item \textbf{Credentials:} For this academic project, credentials are stored in
        source code. Production systems should use environment variables or a secrets
        manager.
\end{itemize}

\section{Backup \& Recovery Plan}

\begin{itemize}
  \item Daily full logical backups using \texttt{mysqldump}, stored with timestamp.
  \item Binary logging enabled for incremental backup and point-in-time recovery.
  \item Backup files stored off-site or in cloud object storage (e.g., AWS S3).
  \item Monthly restore drills to verify backup integrity and recovery time.
\end{itemize}

\begin{lstlisting}[style=sqlstyle, caption={Daily backup command}]
mysqldump -u root -p LibraryDB > LibraryDB_$(date +%Y%m%d).sql
\end{lstlisting}

\section{Performance Optimization}

\begin{itemize}
  \item Indexes on \texttt{BookName} and \texttt{AuthorID} reduce full-table scans.
  \item Views pre-join commonly accessed multi-table data, reducing query complexity.
  \item Stored procedures execute multi-step logic server-side, minimizing round trips.
  \item Single persistent connection avoids repeated connection overhead.
\end{itemize}

\section{Limitations}

Despite achieving the main objectives, the system still has several limitations:

\begin{enumerate}
  \item The application is implemented as a command-line interface (CLI), which
        may not provide an optimal user experience compared to modern graphical or
        web-based systems.
  \item The system does not include an authentication or authorization mechanism.
        All users are assumed to have equal access privileges, which is not suitable
        for real-world deployment where role-based access control is required.
  \item Concurrency control and transaction conflicts have not been thoroughly
        tested. In a multi-user environment, simultaneous borrowing or returning
        actions may lead to inconsistencies without proper isolation handling.
  \item The system does not support advanced features such as book reservation,
        fine calculation for overdue returns, or real-time notifications. These
        features are important in practical library systems and could be considered
        for future development.
\end{enumerate}

\section{Recommendations for Future Improvements}

\begin{enumerate}
  \item Add a \texttt{DueDate} column to \texttt{Borrowing} and implement an automatic
        fine calculation function based on days overdue.
  \item Implement full-text search using a \texttt{FULLTEXT} index on
        \texttt{Books(BookName)} for more flexible catalog search.
  \item Migrate \texttt{src/main.py} to a web-based interface (e.g., Flask + HTML)
        or desktop GUI (e.g., Tkinter).
  \item Add role-based authentication at the application level, mapping login credentials
        to MySQL user roles (librarian vs.\ administrator).
  \item Implement an audit log table populated by triggers to track all DML operations
        with timestamps and user context.
  \item Add a reservation system allowing readers to reserve out-of-stock books.
\end{enumerate}
